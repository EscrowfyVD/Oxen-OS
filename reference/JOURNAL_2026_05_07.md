## Décisions structurelles importantes

### 1. Cron infrastructure — pivot Railway → GitHub Actions

**Tentative initiale** : créer 2 services Railway Cron (`oxen-os-cron-conference-brief` et `oxen-os-cron-signal-decay`) avec build Next.js complet à chaque run.

**Échec** :
- Build Next.js complet inutile pour un script TypeScript de 5 lignes
- DB networking issue (P1001 Can't reach `postgres.railway.internal`) malgré DATABASE_URL en Reference Variable
- Conflits config-as-code entre `railway.json` du repo et UI overrides
- Logs Railway tronqués sur `prisma migrate deploy`, debug opaque

**Pivot** : GitHub Actions workflows.

**Avantages** :
- Pas de build (juste curl HTTP vers les endpoints en prod)
- Free tier (32 runs × 30s = 16 min/mois, well under 2000-min limit)
- Configuration declarative versionned dans le repo (`.github/workflows/*.yml`)
- Logs clairs (curl output + HTTP code)
- `workflow_dispatch` pour manual trigger + replay

**Caveat documenté** : GitHub Actions schedule peut dériver de 15-30 min pendant les heures de pointe (officiel). Acceptable pour cadence mensuelle (Conference Brief) et quotidien off-peak 03:00 UTC (Signal Decay).

## Leçons apprises

### 1. Secret leaks in chat — 2 occurrences en 2 jours

**What** : Vernon a collé dans le chat Claude :
- Hier (2026-05-06) : `SIGNALS_INGESTION_SECRET` (Sprint S1 smoke test) → rotated
- Aujourd'hui (2026-05-07) : `CRON_SECRET` (faux exemple par accident) → vérifié non-réel

**Why** : copier un curl complet du terminal vers le chat inclut le secret expansé en plain text si la variable shell est inline.

**Fix appliqué** :
- Pattern `NEW_SECRET="..."` dans terminal puis `curl -H "Authorization: Bearer $NEW_SECRET"` (variable shell, pas la valeur)
- `echo "Length: ${#NEW_SECRET}"` pour vérifier sans exposer
- Règle interne : ne jamais coller dans le chat des strings qui ressemblent à un secret, même tordues

**À retenir** : tous les secrets HTTP endpoints Oxen OS sont rotatables en 5 min via Railway dashboard si jamais re-leak.

### 2. Sprint Lemlist hardening — bouton "Push 50" alors que body dit "Push 597"

**What** : Sprint S0.6 a fixé le pre-fetch des contacts (footgun #1) mais a oublié le label du bouton (drift entre `contactCount` page-50 et `totalFilteredCount` filtered-all). Détecté en E2E manuel post-deploy.

**Why** : Sprint S0.6 a ajouté `totalFilteredCount` plumbing mais n'a updated que 2 occurrences sur 3 dans le JSX.

**Fix appliqué** : Hotfix S0.6.1 — extract `displayCount` const en haut du render comme single source of truth, harmonise les 3+ occurrences.

**À retenir** : pour les sprints UI, faire un E2E manuel COMPLET avec **chaque bouton cliqué + chaque label vérifié** avant d'appeler le sprint "validé". C'est le rôle d'un PM, lequel arrive bientôt selon roadmap.

### 3. Railway Cron Services build Next.js complet — architecturalement absurde

**What** : tentative initiale d'utiliser Railway Cron Services natifs. Build Next.js de 191 routes pour exécuter un script TypeScript de 5 lignes. Échec après ~30 min de debug (DB networking + config-as-code conflicts).

**Why** : assumption que "tout sur Railway pour cohérence ops" était la bonne archi. En pratique : friction multiple (Build Command, Start Command, Cron Schedule, Restart Policy, Variables, config-as-code path). Railway Cron Services sont mieux pensés pour des **services qui partagent partiellement le code base** (genre worker queue), pas pour des **scripts CLI standalone** qui curl un endpoint.

**Fix appliqué** : pivot GitHub Actions. 30 lignes de YAML, free tier, logs clairs, versionned dans le repo.

**À retenir** : **GitHub Actions est le bon outil pour un cron qui curl un endpoint**. Railway Cron Services pour des scripts qui **doivent** runner dans le container Railway (genre accès DB privé sans endpoint HTTP).

### 4. Audit pré-sprint = ROI massif

**What** : sur Sprint Quick Wins Cleanup, l'audit pré-sprint (5-10 min) a révélé un finding majeur : "data quality 35 contacts UK/US/EU dans G1-T1" était en fait un **bug de segmentation côté UI** (utilise `contact.country` au lieu de `company.country`), pas un bug data.

**Why** : sans audit, on aurait passé 1-2h à exclude/downgrade ces 36 contacts à tort.

**Fix appliqué** : audit a redirigé le sprint vers la vraie cause (rename filtre, switch query Prisma sur la relation Company). Le data reste intact, le filtre fonctionne maintenant correctement.

**À retenir** : **audit pré-sprint = obligatoire** sur tout sprint qui touche à la data. 5-10 min qui économisent des heures.
